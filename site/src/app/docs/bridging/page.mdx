# Cross-Chain Bridging

GasStorm supports cross-chain ETH bridging between L1 and L2 via [Hyperlane](https://www.hyperlane.xyz/). Three L1 modes are available, all sharing the same bridge infrastructure.

## L1 Modes

| Mode | L1 Node | Command | Use Case |
|------|---------|---------|----------|
| **Anvil** | Local Anvil (chain 31337) | `make run-with-bridge` | Default dev/test |
| **Besu** | Local Besu (chain 1337) | `make run-besu-reth-hyperlane` | Besu integration demo |
| **External** | Any remote node | `make run-external-l1-hyperlane` | Connect to existing infra |

## Architecture

```
L1 (Anvil / Besu / External)
  |
  +-- op-reth (L2, chain 42069) -- block-builder -- load-generator -- dashboard
  |
  +-- Hyperlane Bridge
  |     +-- hyperlane-init      (deploys contracts to both chains)
  |     +-- hyperlane-relayer   (relays messages between chains)
  |     +-- hyperlane-bridge-ui (web UI for bridging ETH)
  |
  +-- Block Explorer
        +-- L1 Explorer (port 18202/18203)
        +-- L2 Explorer (port 18200/18201)
```

All modes use the same Hyperlane contracts, relayer, bridge UI, and block explorers. The only difference is what serves the L1 RPC.

## Quick Start

### Anvil L1 (Default)

```bash
make run-with-bridge
```

Starts Anvil as L1 with Hyperlane bridge and bridge UI. No explorers by default.

### Besu L1

```bash
make run-besu-reth-hyperlane
```

Replaces Anvil with Hyperledger Besu (Clique consensus, chain ID 1337, 5-second blocks). Includes bridge, both L1/L2 block explorers.

<Callout variant="info" title="Drop-in replacement">
Besu uses a Docker Compose overlay (`docker-compose-besu-l1.yaml`) that replaces the `l1` service while keeping the same service name. All downstream services work automatically.
</Callout>

### External L1

```bash
EXTERNAL_L1_RPC=http://your-besu-node:8545 \
EXTERNAL_L1_KEY=0xYourFundedPrivateKey \
  make run-external-l1-hyperlane
```

Connects to any remote EVM node. An nginx reverse proxy maps internal `http://l1:8545` to the external URL so all services work unchanged.

<Callout variant="warning" title="Funded key required">
The `EXTERNAL_L1_KEY` must have ETH on the external chain. The init container deploys 4 Hyperlane contracts and funds warp routes. Chain ID and domain are auto-detected from the RPC.
</Callout>

## How It Works

### Contract Deployment

The `hyperlane-init` container deploys Hyperlane contracts to both chains using Foundry:

1. **TestISM** - Interchain Security Module (trusted relayer model)
2. **TestHook** - Merkle tree hook for message dispatch
3. **TestMailbox** - Core Hyperlane mailbox (message send/receive)
4. **HypNativeSimple** - Warp route for native ETH bridging

After deployment, routers are enrolled to connect the warp routes on both chains. The warp routes are funded with 100 ETH each to cover bridge transfers.

### Relayer

The Hyperlane relayer (`gcr.io/abacus-labs-dev/hyperlane-agent:agents-v1.3.0`) watches both chains for dispatched messages and relays them cross-chain. Config is auto-generated by the init container.

### Bridge UI

The bridge UI (port 18001) is a customized Hyperlane Warp UI. Connect any wallet to bridge ETH between L1 and L2.

### Dashboard Integration

The GasStorm dashboard (port 18000) automatically detects the L1 client via `web3_clientVersion` and shows the correct name (e.g., "L1 (Besu)", "L1 (Anvil)"). Bridge relayer status and active transfers are shown in the component status panel.

## Besu L1 Details

### Genesis Configuration

Besu uses a Clique consensus genesis (`genesis/besu-dev-genesis.json`):

| Parameter | Value |
|-----------|-------|
| Chain ID | 1337 |
| Consensus | Clique (single signer) |
| Block Period | 5 seconds |
| Gas Limit | 30,000,000 |
| Signer | Anvil Account #0 (`0xf39F...2266`) |
| Pre-funded | Accounts #0, #1, #2 (100,000 ETH each) |

The signer key is stored in `genesis/besu-node-key` (well-known Anvil test key).

### Solidity Scripts

The Foundry deployment scripts are chain-agnostic:

- `script/DeployBridge.s.sol` - Reads `LOCAL_DOMAIN` from env (defaults to `block.chainid`)
- `script/EnrollRouters.s.sol` - Reads `LOCAL_WARP`, `REMOTE_WARP`, `REMOTE_DOMAIN` from env

No hardcoded chain IDs. Works with any EVM chain.

### Deploy Script Parameterization

`hyperlane-init/deploy.sh` accepts these env vars (with backward-compatible defaults):

| Variable | Default | Description |
|----------|---------|-------------|
| `L1_CHAIN_NAME` | `l1local` | Hyperlane chain name for L1 |
| `L2_CHAIN_NAME` | `l2local` | Hyperlane chain name for L2 |
| `L1_DOMAIN` | `31337` | Hyperlane domain ID for L1 |
| `L2_DOMAIN` | `42069` | Hyperlane domain ID for L2 |
| `L1_CHAIN_ID` | `31337` | L1 chain ID (for relayer config) |
| `L2_CHAIN_ID` | `42069` | L2 chain ID (for relayer config) |
| `L1_BLOCK_TIME` | `12` | L1 estimated block time (seconds) |
| `L2_BLOCK_TIME` | `1` | L2 estimated block time (seconds) |
| `DEPLOYER_PRIVATE_KEY` | Anvil #1 | Key used for contract deployment |
| `AUTO_DETECT_CHAINS` | `false` | Auto-detect chain IDs from RPCs |

## External L1 Details

### How the Proxy Works

When using `docker-compose-external-l1.yaml`, the `l1` Docker service is replaced with an nginx reverse proxy:

```
Internal services --> http://l1:8545 --> nginx proxy --> EXTERNAL_L1_RPC
```

All services (explorer, bridge init, relayer, dashboard) connect to `http://l1:8545` as usual. The proxy transparently forwards to the external URL.

### Environment Variables

| Variable | Required | Description |
|----------|----------|-------------|
| `EXTERNAL_L1_RPC` | Yes | External L1 JSON-RPC URL |
| `EXTERNAL_L1_KEY` | Yes | Private key with ETH on the external L1 |
| `EXTERNAL_L1_CHAIN_NAME` | No | Hyperlane chain name (default: `externall1`) |

### What Gets Auto-Detected

When `AUTO_DETECT_CHAINS=true` (set automatically in external mode):
- **Chain ID** via `eth_chainId`
- **Domain ID** set equal to chain ID
- **Client name** via `web3_clientVersion` (shown in dashboard)

## Service Ports

| Service | Port | Description |
|---------|------|-------------|
| L1 RPC (Anvil/Besu/Proxy) | 18545 | L1 HTTP JSON-RPC |
| L1 WebSocket | 18549 | L1 WS (Besu mode only) |
| Bridge UI | 18001 | Hyperlane Warp UI |
| Bridge Relayer Metrics | 19090 | Prometheus metrics |
| L1 Explorer API | 18202 | Block explorer REST API |
| L1 Explorer UI | 18203 | Block explorer web UI |
| L2 Explorer API | 18200 | Block explorer REST API |
| L2 Explorer UI | 18201 | Block explorer web UI |

## Troubleshooting

### Besu stuck at block 0

If Besu logs show `withdrawals must not be null when Withdrawals are activated`, the genesis has Shanghai enabled but Clique does not support withdrawals. Remove `shanghaiTime` from `genesis/besu-dev-genesis.json`.

### Hyperlane init fails to deploy

Check the init container logs: `docker logs gasstorm-hyperlane-init`. Common causes:
- L1 node not producing blocks (Besu signer key mismatch)
- Deployer account has no ETH (external L1 mode)
- RPC not reachable from Docker network

### External L1 proxy not connecting

Verify the external RPC is reachable from inside Docker:
```bash
docker exec gasstorm-l1 wget -qO- --post-data='{"jsonrpc":"2.0","method":"eth_chainId","params":[],"id":1}' \
  --header='Content-Type: application/json' http://localhost:8545
```

### Bridge UI shows wrong chain names

The bridge UI bakes chain names at build time. Rebuild with:
```bash
docker compose -f docker-compose.yml -f docker-compose-besu-l1.yaml \
  --profile reth --profile bridge up --build -d hyperlane-bridge-ui
```
