## High-Confidence Findings (Found by Multiple Models)

### 🔴 CRITICAL: FCU Set Head Failure Doesn't Block Success
**Found by:** grok-code, minimax (2/3 models)
**Opus Validated:** ✅ CONFIRMED

**Location:** `block-builder/builder.go:811-837`

**Issue:** When FCU set head fails (line 811), the code logs a warning but continues. The critical bug is that `blockSuccess = true` is set unconditionally on line 837, even after FCU failure. This means:
- Transactions appear "confirmed" but may not be in canonical chain
- The defer won't re-queue them for retry
- Potential nonce desync between builder and actual chain state

**Evidence:**
```go
// Line 811-817: FCU fails, just logs
if err != nil {
    log.Printf("WARNING: FCU set head failed...")
}

// Line 837: Set unconditionally!
blockSuccess = true  // BUG: This runs even if FCU failed
```

**Fix:**
```go
if err != nil {
    log.Printf("WARNING: FCU set head failed...")
    blockSuccess = false  // Add this line
    return fmt.Errorf("FCU set head failed: %w", err)
}
```

### 🔴 HIGH: WebSocket Memory Leak - isConnecting Not Reset
**Found by:** minimax, big-pickle (2/3 models)
**Opus Validated:** ✅ CONFIRMED

**Location:** `dashboard/src/stores/load-test-websocket.ts:221-231`

**Issue:** `disconnect()` does not reset `isConnecting = false`. If disconnect is called during a connection attempt, subsequent reconnects will be blocked.

**Evidence:**
```typescript
const disconnect = () => {
    shouldReconnect = false;
    // Missing: isConnecting = false;
    if (reconnectTimeout) { ... }
    if (ws) { ws.close(); ws = null; }
};
```

**Fix:** Add `isConnecting = false;` in disconnect().

---

### 🔴 HIGH: WebSocket rafId Not Cancelled on Disconnect
**Found by:** grok-code, minimax (2/3 models)
**Opus Validated:** ✅ CONFIRMED

**Location:** `dashboard/src/stores/load-test-websocket.ts:221-231`

**Issue:** `rafId` (requestAnimationFrame) is not cancelled in disconnect(), allowing stale callbacks to execute after disconnect.

**Fix:** Add `if (rafId) { cancelAnimationFrame(rafId); rafId = null; }` in disconnect().

### 🟡 HIGH: Nonce Rollback Logic Issue
**Found by:** grok-code, big-pickle (2/3 models)
**Opus Validated:** ❌ FALSE POSITIVE

**Location:** `load-generator/internal/account/account.go:64`

**Issue Claimed:** Race condition in Nonce.Rollback() - Swap returns true if committed was false.

**Actual Behavior:** The code is correct. `Swap(true)` returns the **old** value, not the new one. If committed was already true (committed or already rolled back), Swap returns true and we skip rollback. If committed was false, Swap returns false and we DO rollback. This is correct.

### 🟡 MEDIUM: BulkInsert All-or-Nothing Rollback
**Found by:** grok-code, minimax, big-pickle (3/3 models)
**Opus Validated:** ⚠️ Design Decision

**Location:** `load-generator/internal/storage/sqlite.go:519-555`

**Issue:** If any single insert fails, `defer tx.Rollback()` discards ALL rows.

**Analysis:** This is actually intentional atomic behavior. For data integrity, all-or-nothing may be preferred. However, for resilience in load testing, partial success might be more useful.

---

### 🟡 MEDIUM: Timer Drain Race Pattern
**Found by:** minimax
**Opus Validated:** ✅ VALID (common Go pitfall)

**Location:** `block-builder/production.go:124-129`

**Issue:** Timer.Stop() may miss timer.C between drain check.

**Fix:**
```go
if !waitTimer.Stop() {
    select {
        case <-waitTimer.C:
        default:
    }
}
```

## Recommendations

### Immediate Fixes (Critical)
1. **FCU set head failure handling** - Set `blockSuccess = false` on FCU failure
2. **WebSocket disconnect cleanup** - Reset `isConnecting` and cancel `rafId`
3. **SQLite PrepareContext rollback** - Add explicit `tx.Rollback()` on prepare failure

### Short-term Fixes (High)
1. Add mutex protection in `collectPendingTransactions`
2. Fix error channel buffering in `InitializeNonces`
3. Use correct client for nonce sync in `FundDynamicAccounts`
4. Add circuit breaker for Engine API calls

### Technical Debt (Medium)
1. Implement bounded timeSeries arrays in dashboard
2. Add context cancellation checks in bulk operations
3. Consider nonce gap tracking for better rollback handling
